# Phase 3: 撰写

规划文章结构，逐章写作，去 AI 味审查，最终输出。

---

## 内联规则 A：文章风格（来自 technical-writing）

### 核心原则

- **逻辑缜密**：清晰结构，层级分明，逻辑流畅
- **叙事感**：像讲故事一样递进式解释概念
- **直接高效**：不说废话，尊重读者时间
- **第一人称**：用"我"分享观察和经验
- **偶尔口语化**："说白了"、"说实话"点缀，不滥用

### 文章结构

```
1. 开头 (1-2 段)
   - 背景：为什么这个主题现在值得关注
   - 引子：个人观察或连接点
   - 可选：轻松表达

2. 目录
   - 6-8 个主要章节
   - 标题简洁（≤ 10 字）

3. 核心内容（每章）
   - 概念定义（清晰精确）
   - 技术细节 + 具体例子
   - 对比（用表格）
   - 实践洞察
   - 优缺点评估

4. 总结
   - 3-5 条可操作的 takeaways
   - 不写"综上所述"
```

### 语气

- 专业但对话感
- 自信、有见地
- 偶尔轻松幽默
- 批判性评估（优缺点都说）

### 语言

- 精确的技术术语
- 清晰简洁的句子
- 避免学术腔被动语态
- 少量口语化表达强调重点

### 用户学到的约束

- 改写类任务保持与原始参考文档核心目标一致
- 写技术文章要添加信息来源（链接和出处）
- 过度添加个人叙事和口语化会使文档冗余
- 不要为了应用写作风格而偏离原始目的

## 内联规则 B：去 AI 味（来自 write-zh-prose）

### 最高优先级：保语义 > 去 AI 味

先确认事实、逻辑、因果不变，再做以下改写。如果为"更口语"改坏原意，属于失败改写。

### 高频 AI 痕迹（写完每章后立即检查）

1. **段末收尾总结句**："到这里"、"这说明"、"这本身就是"、"可以看出"开头的短抽象句。识别信号：段落最后一句比其他句短且抽象。直接删。

2. **升华句**：把具体工程观察上升到普适人生道理。直接给出具体建议，不升华。

3. **对比句式**："不是…而是…"、"X 本身没有价值，真正有用的是 Y"、"X 已经不是瓶颈，Y 才是"。直接说结论，不用对比框架铺垫。

4. **bold 小标题模式**：`**xxx**。content` 改为 `**xxx**，content`，让 bold 变成承重词，不是独立小标题。例外：bold 部分本身是完整句子时保留句号。

5. **章节引介过渡句**：章节开头或结尾写"上面这些模式解决的是…，下面再看…"。直接删掉，章节标题本身已经承接。

6. **空泛形容词预判**："更干净："、"逻辑很清晰："，形容词抢先下判断。删掉形容词，只留后面的事实。

7. **工整并列 bold 标题**：四个 bold 标题全是同一格式。每个点语气不同，或改成散文叙述。

8. **段内重复**：同一段把同一意思用不同措辞说两遍。说一遍。

9. **讲解腔起手**："真拆开看"、"这背后是同一个变化"、"真正关键的问题是"。删掉，直接说。

10. **训人感起手**："先问问是不是"、"你要先明白"。改成"可以先看"、"先确认"。

### 句式规则

- 禁止用破折号（—），用逗号或分号代替
- 四五个句号连发、每句很短 = 打电报，合成长句或用逗号连
- 单独成段的一两句话，多半是上一段的收尾，直接并进去
- 不用"首先…其次…最后"，用"一方面…另一方面…"

### 用词去正式化

| 不用 | 用 |
|------|-----|
| 非常/极其 | 很 |
| 综上所述 | 直接收尾 |
| 例如 | 比如 |
| 购买/使用 | 买/用 |
| 值得注意的是 | 直接说结论 |

## 内联规则 C：可选 Q-A 预处理（来自 ljg-qa）

写章节前，可以对每个章节做 Q-A 骨架提取，帮助理清写作逻辑。

### Q 的四类（至少混合两类）

| 类型 | 模式 |
|------|------|
| 动作 | "它是怎么做到的？" |
| 对比 | "为什么是 A 不是 B？" |
| 因果 | "为什么这个解法成立？" |
| 边界 | "它在什么时候失效？" |

### Q 的禁忌

- 禁止 "什么是 X？"（一句定义打发）
- 禁止 "X 有几个步骤？"（问目录）
- 禁止 "X 重要吗？"（答案预设）

### Q 的语气

口语 > 学术腔。20 字以内 > 长复合句。

### A 的结构（可选使用）

如果使用 Q-A 预处理，每个 A 包含：
1. **结论**：一句话，能脱离上下文被抄走
2. **形式化**：一行可视关系式（如 `A = B + C`）
3. **怎么想到的**：2-4 条短句，每条只走一步推理
4. **边界**：不成立的条件

### 何时使用

- 章节内容复杂、有多个相互关联的机制 → 使用 Q-A 预处理
- 章节内容直接、逻辑清晰 → 跳过，直接写

## Step 1: 确认章节规划

基于 Phase 2 输出的章节规划，最终确认章节结构：

- 每章标题（简洁，≤ 10 字）
- 每章一句话概述要讲什么
- 章节之间的逻辑依赖（哪章铺垫了哪章）

**可选终章"工程迁移"**: 如果 Phase 2 的 5.6 节输出了有效的迁移素材（非"迁移素材不足"），向用户建议在末尾加一章，内容是从这个项目中提炼的可复用设计模式和架构公式。如果迁移素材不足，跳过此章。

向用户展示章节规划，等用户确认后开始写作。

## Step 2: 逐章写作

按章节顺序写作。每章的写作流程：

### 2.1 回顾相关素材

从 Phase 2 的知识地图中找出与本章节相关的：
- 核心机制描述
- 源码细节和代码片段
- 文档 vs 源码差异
- 非显而易见的发现

### 2.2 可选：Q-A 骨架提取

如果本章节内容复杂（3+ 个相互关联的机制），先做 Q-A 预处理：
- 提出本章节要回答的 3-5 个核心问题
- Q 类型混合至少两类
- 按依赖关系排序（Q1 答完 Q2 自然冒出来）

如果内容直接清晰，跳过此步。

### 2.3 撰写章节

写作时遵守：

- **每个技术论断有据可查**：引用源码文件名、行号、或配置值
- **源码优先**：文档与源码不一致时以源码为准，标注"文档 vs 源码差异"
- **具体 > 抽象**：用代码片段、配置值、数据流说明，不空泛描述
- **对比用表格**：多维度对比用 Markdown 表格
- **机制用流程**：复杂机制用步骤式或流程式说明
- **章节结尾不写总结句**（AI 味 #1）
- **枚举覆盖深度**：当章节包含 N 个并列项（如 10 层、4 阶段、3 种模式），每个项必须独立成段，至少包含：用途（1 句）、实现机制（2-3 句，含函数名/常量名）、设计决策（1-2 句，为什么这样设计）、具体证据（代码片段或配置值）。如果分析阶段的素材不足以支撑某个项的深度，在该项末尾标注 `[素材不足，待补充]` 而不是用一句话带过。不同项不合并到同一段落（除非两个项高度耦合——由同一个函数处理且逻辑完全交织）

### 2.4 可选终章：工程迁移

如果用户确认包含此章，基于 Phase 2 的 5.6 迁移素材撰写。结构：

**可偷的设计模式**（2-3 个，每个）：
- 模式名 + 一句话定义
- 这个项目怎么用的（具体到模块和函数）
- 在什么其他场景下也能用
- 如果有架构公式，附上

**避坑清单**（0-2 条，如果有的话）：
- 反模式描述 + 为什么别这么干

**写作要求**：
- 不搞升华，不搞认知跃迁，只讲工程事实
- 每个模式必须有源码证据，不能凭空编
- 如果素材只有 1 个有效模式，就只写 1 个，不凑数
- 章节长度控制在其他章节的一半左右，精简为主

如果 5.6 输出为"迁移素材不足"，此章降级为一段简短的"总结与启发"（3-5 条 takeaway，不单独成章，合并到现有总结章节）。

### 2.5 章节内去 AI 味

写完每章后，立即检查 10 条 AI 痕迹 + 句式规则 + 用词规则。发现问题立即修改，不攒到最后。

## Step 3: 全文去 AI 味审查

所有章节写完后，做一次全文扫描。逐条过：

**10 条 AI 痕迹**：
- [ ] 没有段末收尾总结句
- [ ] 没有升华句
- [ ] 没有对比句式框架（"不是…而是…"）
- [ ] bold 是承重词不是独立小标题
- [ ] 没有章节引介过渡句
- [ ] 没有空泛形容词预判
- [ ] bold 标题语气各异
- [ ] 没有段内重复
- [ ] 没有讲解腔起手
- [ ] 没有训人感起手

**句式与用词**：
- [ ] 没有破折号
- [ ] 没有打电报式短句连发
- [ ] 没有"首先…其次…最后"
- [ ] 用词已去正式化

**语义完整性**：
- [ ] 改写后事实、逻辑、因果不变
- [ ] 每个技术论断仍有证据支撑

任何一条不过，回去改。改完重新过这条。

## Step 4: 写入文件

获取时间戳：`date +%Y-%m-%d`

输出路径和文件开头的 YAML front matter 格式见 [SKILL.md 的输出规范](../SKILL.md)。如果用户指定了 `output_dir`，优先使用用户路径。

用 Write 工具写入完整文件。写入后用 Read 工具确认文件已正确保存。

## Step 5: Vault 集成建议

检查当前环境是否安装了 `loom-maintain` skill。判断方式：在可用 skill 列表中查找 `loom-maintain`。

**如果存在 loom-maintain**：向用户建议运行 loom-maintain 处理 vault 集成（创建 Topic Note、更新索引、发现跨主题连接）。不需要 source-dive 自己操作 vault。

**如果不存在**：跳过此步，不做任何 vault 操作。source-dive 的输出是自包含的文章文件，不依赖 vault 结构。
