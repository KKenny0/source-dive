# Phase 2: 分析

并行分析源码，提取架构模式和关键机制，构建知识骨架。

## Step 1: 准备分析

回顾 Phase 1 的发现摘要。将文件按大小分为两组：

- **主线程组**（small 文件）：直接在主线程读取分析
- **并行组**（medium/large 文件）：启动 background agent 并行分析

## Step 2: 启动并行分析

对每个 medium/large 文件，启动一个 background agent。

**关键**：多个 background agent 必须在一次消息中并行启动（多个 Agent tool use），不要串行。

给每个 agent 的 prompt 模板：

```
分析 GitHub 仓库 {repo_name} 中的文件 {file_path}。

用 mcp__zread__read_file 读取文件内容（如果文件很大，分段读取）。

提取以下信息：

1. **核心类/函数**: 列出最重要的类和函数（最多 10 个），每个用一句话说明职责
2. **关键机制**: 这个文件实现了什么核心机制？怎么工作的？用 2-3 句话说清楚
3. **与其他模块的交互**: 它 import 了谁？谁可能 import 它？列出关键依赖
4. **设计模式**: 用了什么设计模式？（如单例、工厂、观察者、策略等）为什么这样设计？
5. **硬编码常量和配置**: 列出关键常量、阈值、默认值
6. **非显而易见的细节**: 有什么从文档里看不出来、只有读代码才能发现的？
7. **枚举项逐一分析**: 如果文件实现了枚举式系统（如 N 层 prompt、M 个阶段、K 种模式、P 个平台适配器），对每个枚举项逐一提取：
   - 用途（这个项解决什么问题）
   - 实现机制（哪个函数/常量/配置驱动它）
   - 触发条件或生效范围（什么时候被包含/排除）
   - 与其他项的关键差异（不要把所有项混在一起描述）
   - 具体证据（函数名、配置值、关键代码片段）

输出要求：
- 每个点 2-3 句话说清楚
- 枚举项（第 7 点）每个项至少 3-5 句，不要合并不同项到同一段
- 不要粘贴大段代码，只引用关键行
- 中文输出
```

## Step 3: 主线程分析小文件

在等待 background agents 完成的同时，主线程直接读取和分析 small 文件：

1. 用 `mcp__zread__read_file` 读取文件内容
2. 提取：
   - 核心类名和继承关系
   - 关键函数签名和职责
   - 配置常量和默认值
   - import 依赖（看出模块交互关系）
3. 记录到临时分析笔记中

**分析重点**：
- 这个模块在整个系统中的角色是什么？
- 它解决了什么问题？
- 有没有非显而易见的实现细节？

## Step 4: 收集 background agent 结果

Background agents 完成后，逐个收集结果。

每个 agent 的结果整理为：
- 核心发现（2-3 条）
- 关键机制描述
- 与其他模块的关系
- 文档未提及的实现细节

**注意**：如果某个 agent 的分析不够深入（比如因为文件太大只读了部分），标记该文件需要补充分析。

## Step 5: 构建知识地图

汇总所有分析结果（主线程 + background agents），构建知识地图：

### 5.1 架构概览

用一段话描述整体架构。画一个简单的模块关系图：

```
{模块A} ←→ {模块B}
   ↓           ↓
{模块C} ←→ {模块D}
```

### 5.2 核心模块清单

每个模块一行：
- `{module_name}`: {职责} → 依赖 {other_modules}

### 5.3 关键机制列表

每个机制一段：
- **{机制名}**: {怎么工作的}。为什么这样设计。

### 5.4 文档 vs 源码差异

逐条记录文档描述与源码实现不一致的地方：
- **{差异点}**: 文档说 {X}，源码实际是 {Y}。影响是 {Z}。

### 5.5 非显而易见的发现

只有读源码才能发现的东西：
- **{发现}**: {细节描述}

### 5.6 可迁移模式识别（为终章"工程迁移"准备素材）

从所有分析结果中识别可跨项目复用的工程智慧。输出 0-3 组，不用凑数。

每组包含：

- **模式名**: 一个能被记住的短名（如"插件注册表"、"分层调度器"）
- **这个项目怎么用的**: 具体到模块名和关键函数，2-3 句
- **为什么在这里有效**: 解决了什么问题，不用会怎样
- **适用场景**: 什么类型的其他项目也能用这个模式
- **架构公式**（可选）: 如果模式有清晰的结构组成，压缩成一行（如 `插件系统 = 接口契约 + 注册表 + 加载器 + 生命周期钩子`）

额外记录**避坑点**（0-2 条）：源码中发现的反模式、过度设计、或明显的痛点，标注"别这么干"及原因。

**质量门控**: 如果分析素材不足以提炼出有深度的模式（比如项目本身平庸、或分析不够深入），直接输出"迁移素材不足"，终章将降级为简短总结。不要硬编空泛的模式。

## Step 6: 规划章节骨架

基于知识地图，规划文章的初步章节结构：

1. 每章一个核心主题（一个关键机制、一个架构层面、一组对比等）
2. 章节之间有逻辑依赖：前面章节为后面铺垫
3. 考虑读者的理解路径：从宏观到微观，从概念到实现

输出格式：

```
## 章节规划

1. **{章节名}** — {一句话描述要讲什么}
2. **{章节名}** — {一句话描述}
3. ...
```

数量控制在 6-8 章。

完成后进入 [Phase 3: 撰写](03-compose.md)。
